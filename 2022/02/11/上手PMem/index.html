Part 1 Persistent Memory Provisioning Introduction

参考自：QSG Part 1 Provisioning with Intel® Optane™ PMEM

简介

Intel的Optane DC Persistent Memory支持两种模式：Memory Mode 和 App Direct Mode(，Mixed Mode)。

详情Intel® Optane™ DC Persistent Memory Operating Modes Explained - IT Peer Network

  • IPMCTL是intel官方创建并维护的用于配置和管理Optane DCPMM的开源实用程序，用于管理Region。

  • NDCTL是非易失性设备控制，用于管理Namespace。

持久性内存相关概念

Region

Region是指一组一个或多个NVDIMM，也称为交错集，可以以n路交织或非交织的方式创建Region。 在Intel Optane DC PMM上，只能使用 ipmctl 创建或更改区域。

支持的Region有三种：PMEM，BLK和NVDIMM，Region可以划分为一个或多个namespace（命名空间）。

  • PMEM：持久化内存设备允许字节可寻址访问。

  • BLK：块设备就像传统的存储设备一样允许扇区原子性。

  • NVDIMM：这些设备可以同时支持PMEM和BLK模式访问。

我的理解是，Region就像一个物理磁盘，而其namespace就相当于物理磁盘通过分区划分为多个逻辑磁盘

Namespace

Namespace用于定义一个连续寻址的非易失性存储器，其概念类似于硬盘分区，SCSI逻辑单元（LUN）或NVM Express命名空间。 它是持久性存储器的单元，在/dev中作为可用于I/O的设备出现。 英特尔建议使用非易失性设备控制（ndctl）为Linux操作系统创建名称空间。

DAX

直接访问（DAX）是一种允许应用程序直接从CPU（通过加载和存储）访问持久性介质的机制，绕过传统的I/O堆栈（页面缓存和块层）。

Namespace Types

  • Filesystem-DAX(FSDAX)
  • Device DAX(DEVDAX)
  • Sector
  • Raw

PMem Provisioning Overview

使用PMem的准备工作分两步（Memory Mode只需第一步，AppDirect Mode执行下述两步）

  1. 配置Goal(Region)：在这一步骤中，goal会被识别并存储在PMem modules中，并在下次重启时应用。有三种模式：Memory Mode, App Direct mode, or mixed mode.Goal可以通过以下两者之一的方法设置。
    • Bios设置
    • 操作系统实用程序：IPMCTL
  2. 配置Namespaces：只有App Direct Mode需要。这是 OS-specific 操作。
    • Linux: ndctl utility
    • Windows: Powershell

IPMCTL工具

Ipmctl可以从统一可扩展固件接口（UEFI）shell或操作系统中的终端窗口启动。 两个版本的ipmctl都支持相同的功能，通过从命令行运行ipmctl help 或者 man ipmctl，可以看到完整的命令列表。Linux和Windows环境下的Commands相同。

Tutorial: 如何创建namespace
  1. 首先需要确定我们是要配置系统全部的PMem容量还是其中的一部分。
    • 如果要对全部region创建单一namespace，直接使用：sudo ndctl create-namespace
    • 如果只用到PMem容量的一部分，则指令如：sudo ndctll create-namespace -m fsdax -s 54G这个指令的参数解释如下：
      • Argument Details
        • fsdax: Namespace mode
        • 54G: capacity(G!=GiB)
      • Argument Results
        • 创建了一个新的/dev/pmem{X[.Y]}设备
        • X.Y代表这是第X个Region上的第Y个namespace
  2. 输入指令ls -l /dev/pmem*来确定namespace成功被创建
  3. （略）使用指令 ndctl create-namespace -n "MyApp"将namespace标记为”MyApp”
  4. 使用指令sudo ndctl create-namespace来指定剩余全部PMem容量为一个namespace
  5. 使用指令ndctl list -N列出全部创建的namespace
  6. 使用指令如mkfs.ext4 -f /dev/pmem1来创建文件系统并挂载到PMem设备上
  7. 使用指令如 mkdir /mnt/myPMem来创造挂载点
  8. 使用指令如mount -o dax /dev/pmem1 /mnt/myPMem 来挂载文件系统，通过-o dax选项。

Appendix: 如何将PMem恢复出厂设置

Quick Method

  1. sudo ndctl destroy-namespace -f all
  2. OS> ipmctl delete -f -dimm -pcd'
    1. Deletes the Goal aka PCD (Platform Config Data)
  3. Reboot the system

Long Method

  1. sudo登录
  2. ndctl list -N: Get a list of namespaces
  3. Unmount the file system from the namespace needing to be removed
  4. ndctl disable-namespace: Disable namespaces (repeat steps for all namespaces required)
  5. ndctl destroy-namespace
  6. ipmctl delete -f -dimm -pcd: Delete the Goal aka PCD (Platform Config Data)
  7. Reboot

Getting Started Guide

参考自：Getting Started Guide - Persistent Memory Documentation (pmem.io)

Introduction

PMem综述

Persistent Memory(Non-Volatile Memory)出现的背景是：CPU一直越来越快，core/thread数也越来越多，尽力维持Moore定律，但是DRAM和传统非易失性存储(NAND SSDs， HDD)的容量、价格、速度进步缓慢，成为系统和应用性能的瓶颈。

PMem位于存储结构中DRAM和NAND SSD之间，容量比DRAM大，性能比SSD好，应用可以像对DRAM访问那样访问PMem，消除了在内存和存储器之间来回分页数据块的需要。(?)

PMem对application developers意味着什么？

  • 数据的持久性
    • 数据需要flush后才能持久化，一直以来都是这样，但flush的问题过去没被programmers重视。虽然调用标准API也可以work，但更多种的flush，(where the application flushes stores from the CPU caches directly, instead of calling into the kernel, is also possible.
    • CPUs有乱序执行和cache访问/刷回
  • 数据的一致性
    • PMem可以使得8-byte store to pmem是powerfail atomic。对传统的保证一致性方法（transactions/logging/recovery）带来变革。
  • 内存泄露
    • 对传统持久化存储来说，重启不会改变内存泄露，泄露了的内容重启还是泄露。对目前的易失性内存，重启后会释放掉泄露的内存。
  • 字节级寻址
  • Error Handling
